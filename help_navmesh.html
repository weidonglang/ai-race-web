导航网格与路径规划原理

本页介绍 AI Race 中使用的 导航网格（Navigation Mesh, NavMesh） 概念、烘焙流程， 以及它在 编辑器 与 Arena 场景中的实际用途。

可以把 NavMesh 理解成：在复杂 3D 场景中，为智能体预先计算好的「可行走区域」与「可通行连接」， 路径规划算法只在这张网格上搜索路径，从而大幅降低运算量。

关键词：NavMesh / 寻路 / 可行走区域 关联页面：editor.html / arena.html

1. 什么是导航网格？

在 3D 游戏或仿真场景中，如果直接在「连续空间」里做路径搜索， 搜索空间非常大（几乎无限），不利于实时计算。 因此我们通常会先对场景进行离散化，得到一张简化的图或网格。

导航网格（NavMesh） 是其中一种常见表示方式：

  • 把可行走区域划分成若干凸多边形（或规则小块），拼成一整张「地毯」。
  • 多边形之间共享边界时，视为可以相互到达，形成一张连通图。
  • 寻路时不再关心每一个三角形顶点，而是只在这些区域中心/边界上搜索路径。

好处是：

  • 搜索点数远小于网格世界的所有格子，运算更快。
  • 能更好地贴合真实几何形状，比简单的规则网格更精细。
  • 可以在 NavMesh 上叠加额外信息，例如区域权重、危险度等。

2. 本项目中的 NavMesh 思路

在 AI Race 中，Arena 和 Editor 使用的是一个简化版导航网格 思路：

  • 场景地面被划分为一系列规则的小块（可以理解为「地板瓦片」）；
  • 每个小块标记为：可走（walkable）或 不可走（blocked）；
  • 所有可走小块按相邻关系（上下左右 / 斜对角）连成一张图，这就是我们的「NavMesh」；
  • 智能体在网格上移动，同时用 3D 模型在场景中渲染出来。

与传统引擎里的「不规则多边形 NavMesh」相比，本项目的实现更接近于 规则网格上的导航图，但理念完全一致：预先标记可走区域，再在图上寻路。

3. 在编辑器中的烘焙流程

在 编辑器（editor.html） 中，我们可以通过鼠标交互来设计场景：

  • 添加 / 删除墙体、障碍物；
  • 设置起点、终点和检查点；
  • 选择不同的布局 preset。

当场景发生变化时，内部会执行一个简化的「烘焙（Bake）」过程：

  1. 遍历场景中所有网格小块，检查是否与障碍物重叠。
  2. 为每个小块赋值：walkable = true/false。
  3. 对于 walkable = true 的小块，建立与邻居之间的连边。
  4. 最终生成一份「NavMesh 数据」，供 Arena 和 RL 逻辑使用。

可以把这个过程理解为：先在编辑器里布置迷宫，再点击「生成」， 后台帮你把能走的格子全部算一遍。

4. Arena 中的路径规划

在 Arena（arena.html） 中，导航网格会被实际用于两类路径规划：

4.1 传统寻路（例如 A*）

当需要为「幽灵选手」或调试路径时，可以在 NavMesh 图上调用 A* 等经典算法：

  • 状态：每一个 可走格子；
  • 邻居：上下左右（以及可能的对角）可走格子；
  • 代价：通常为 1 或对角线 √2；
  • 启发函数：曼哈顿距离 / 欧氏距离等。

算法输出的是一串格子坐标，随后会被映射回 3D 空间中的路径点， 供小球沿着路径平滑移动。

4.2 强化学习中的「环境骨架」

在强化学习部分，智能体虽然是通过试错来学策略，但它依然离不开 NavMesh：

  • 只有在 NavMesh 上标记为可走的格子，才会被视作合法状态；
  • 格子之间的相邻关系，决定了「上、下、左、右」动作是否有效；
  • 撞到墙体、本不该存在的状态，会被判定为非法并给出惩罚。

可以说：NavMesh 决定了「这个世界能走到哪儿」， 而强化学习只是在这张网里学习「怎么走才更好」。

5. 小结与扩展方向

总结一下，本项目中的导航网格系统起到了两层作用：

  • 为 3D 场景提供清晰的「可走区域」，方便可视化和调试；
  • 为强化学习算法提供规则的离散状态空间，保证训练过程合理、安全。

未来可以继续拓展：

  • 给不同区域设置不同「通行成本」，实现带权 NavMesh；
  • 允许动态障碍物，根据时间重新部分烘焙；
  • 对接更复杂的三角形 NavMesh 数据，实现不规则地图。

建议在课堂展示或报告中，可以配合一张「原始迷宫」与「NavMesh 网格」的对比图， 更直观地说明本节内容。
